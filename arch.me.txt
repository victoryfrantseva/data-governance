Архитектурное описание проекта ККД и Data Lineage

1. Контекст и назначение
Проект dg_full — реализация платформы Data Governance (DG).
Основная СУБД — Аналитическая Платформа (АП) (Greenplum), схема метаданных: dg_full.

Платформа решает две связанные задачи:

Модуль	Назначение
ККД (Контроль качества данных)
Автоматизированные технические и бизнес-проверки данных на слоях STG → ODS → DV → DM, фиксация ошибок, алерты, мониторинг
Data Lineage (DL)
Построение сквозной цепочки зависимостей объектов от источников до дашбордов Qlik Sense, оркестрация потоков, контроль актуальности
Оба модуля опираются на единое хранилище метаданных в виде графа (модель, близкая к GraphDB/DAG).

3.2. Рёбра графа (meta_edge_link)

ODS → DV → DM → QVD / BI
ODS → CTL → DM

4. Архитектура ККД (Контроль качества данных)
4.1. Концепция
ККД выполняется внутри АП в момент или после передачи данных между слоями:

Подготовка метаданных (бизнес-ключи, группы объектов)
Настройка шаблонов проверок
Запуск в составе ETL-потока или отдельным потоком
Сохранение неконсистентных записей и уведомление ответственных
4.2. Метамодель проверок
meta_object_group_ref_table   — группа объектов, проверяемых одним запросом
        ↓
meta_object_ref_table         — конкретные таблицы/поля группы
        ↓
meta_screen_template_ref_table — SQL-шаблон проверки (с алиасами {{..}} или прямой SQL)
meta_screen_template_alias_ref_table — подстановка алиасов
        ↓
meta_screen_link              — привязка проверки к группе, etl_stage, period_run, severity
        ↓
meta_error_event_fact         — результаты выполнения
meta_batch_fact               — пакет запуска
Типы проверок (по документации):

сверка количества записей источник ↔ приёмник
уникальность по бизнес-ключам
заполненность критичных полей
корректность формата (даты и т.п.)
бизнес-проверки (meta_add_business_kkd, категории Incorrect / Ambiguous / Inconsistent / Incomplete)
4.3. Движок выполнения
Ключевая функция — return_meta_run_batch_v2/v3:

Принимает etl_stage, схему/таблицу, wf_load_id, wf_id, period_run
Формирует batch_id, подставляет шаблоны SQL, создаёт temp-таблицы
Использует статистику vmeta_error_event_stat для динамических порогов (avg ± stddev)
Пишет результаты в meta_error_event_fact
Поддерживает режим тестирования шаблонов (v_test_templ)
Запуск может быть:

совместно с загрузкой (по wf_load_id конкретной таблицы)
отдельным потоком (глобальная проверка по всему объёму таблицы)

5. Архитектура Data Lineage
5.1. Назначение
Модуль формирует сквозную цепочку зависимостей от источников (АС, Excel, АП) до приложений BI и решает задачи:

impact analysis («что затронет изменение объекта»)
root cause analysis («откуда пришли данные»)
мониторинг актуальности и duration загрузок
сравнение плановых и фактических зависимостей
5.2. Загрузка метаданных lineage
Источник → Процесс → Приёмник	Способ
Table → View
Автоматически из метаданных GP
Table/View → Function → Table
Автоматически из GP
Table/View → CTL → Table
Ручной / частично автоматизированный; целевой — API CTL
CTL → CTL
Оркестрация потоков; «самоинициализация» потоков
Table/View → BI ETL App → QVD → Dashboard
Ручной
Источник → BI
Нецелевые интеграции (тех. долг)
Текущее состояние: заполнение структур — преимущественно скриптами; целевое — автоматический сбор через API GP, CTL, AirFlow, BI.

«Самоинициализация» потоков: в каждый CTL-поток встраиваются функции init (метаданные потока) и fix (статистика запуска).

5.3. Аналитика потоков
Сравнение плановых связей vs фактического исполнения
Контроль SLA по витринам (meta_scheduler_plan, vmeta_check_kkd_duration)
Аналитика объёма загрузок, отклонения от нормы
Мониторинг актуальности данных АП (отдельные дашборды DL)
5.4. UI: DE control
Инструмент Data Lineage (DE control) в Qlik Sense:

построение lineage по объектам (GraphViz через return_meta_search_graph)
мониторинг работы и обновления объектов

10. Ключевые архитектурные решения
Graph-in-RDBMS — lineage и оркестрация хранятся реляционно (node + edge), обход — рекурсивный SQL; визуализация — GraphViz.
Metadata-driven DQ — проверки декларативны: шаблон SQL + группа объектов + расписание; без изменения кода движка.
Единая схема meta_* — ККД и DL разделяют модель объектов; enable_dq на рёбрах связывает lineage с проверками.
Push-down execution — проверки выполняются в Greenplum рядом с данными, а не во внешнем сервисе.
BI как UI-слой — без отдельного web-приложения; кастомный graph-UI рассматривался как альтернатива.
Версионность и SCD — eff_from_dttm / eff_to_dttm / last_seen_dttm на узлах и рёбрах.
11. Ограничения и технический долг
Ручное поддержание связей Table/View → CTL и BI lineage
Неполная автоматизация сбора метаданных (целевой API-подход не завершён)
Дублирование структур при распределённой архитектуре (риск рассинхронизации)
Нецелевые интеграции (источник → QVD/BI напрямую) отображаются в графе, но помечены как тех. долг
AirFlow как оркестратор — в перспективе, не в текущей реализации
Целевое выделение мета-БД в PostgreSQL — рассматривалось, сейчас всё в схеме АП
Это описание основано на DDL/DML артефактах (ddl_*_20260601.sql, dml/) и документации в docum/. Если нужно, могу детализировать отдельно модель проверок ККД, процесс загрузки lineage или карту функций return_meta_* / fill_meta_*.

2. Слои метамодели
2.1. Шаблон проверки — meta_screen_template_ref_table

Поле	Назначение

ID шаблона (PK)
screen_sql
SQL проверки

Тип данных: Numeric, Text, All и т.д.

Бизнес-описание
Контракт SQL-шаблона — результат должен возвращать ровно два столбца:

SELECT
    JSON_BUILD_OBJECT('unit', 'cnt', 'attr', '{{field_name}}', ...) AS record_identifier,
    <число или текст> AS v_value
record_identifier — JSON-идентификатор строки результата (агрегат, атрибут, детальная запись)
v_value — метрика нарушения (количество, процент, флаг и т.д.)

3.3. По контексту wf_load_id
Движок return_meta_run_batch_v2 принимает:

Параметр	Назначение

Режимы фильтрации в шаблонах ({{p_wf_load_id}}):

Значение	Поведение
Конкретный wf_load_id

-2

Inline с загрузкой — p_tbl_rstri_wf_load_id = p_wf_load_id (проверяем только что загруженные данные)

4. Движок исполнения
4.1. Точка входа

4.2. Алгоритм выполнения
return_meta_run_batch_v2
Создать batch_id
Загрузить статистику порогов temp_meta_error_event_stat
Выбрать object_ref для schema/table
Выбрать screen_link по etl_stage + period_run
Для каждой проверки
Подставить алиасы в screen_sql
CREATE TEMP TABLE temp_out_tbl AS ...sql
Рассчитать severity по порогам
INSERT meta_error_event_fact

5.3. Статистика — meta_error_event_stat

Используется для детекции аномалий («объём загрузки сильно отличается от нормы»)

14. Резюме архитектурных принципов модели
Template → Instance — один шаблон порождает множество экземпляров через screen_link + object_group
Group = один SQL — все объекты группы участвуют в одном запросе
Contract-driven SQL — шаблон обязан вернуть (record_identifier JSON, v_value)
Adaptive thresholds — severity определяется исторической статистикой, не только статическим порогом
ETL-native execution — проверки встроены в CTL-потоки через period_run='ctl'
Layer-aware — разные наборы проверок на STG / ODS / DM
Declarative onboarding — Excel/CSV → JSON → DML, без изменения движка
Если нужно, могу дальше разобрать конкретный сценарий: только STG→ODS, только бизнес-проверки, или формат JSON для генерации DML.
