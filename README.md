# data-governance
Framework of Data Quality &amp; Data Lineage

# Архитектура системы контроля качества данных и построения Data Lineage (Greenplum)

## 1. Общее назначение

Решение построено на **Greenplum** и предназначено для:

- **Управления метаданными** объектов (таблицы, представления, функции, потоки ETL, приложения BI, внешние сущности).
- **Контроля качества данных (ККД)** – автоматизированные проверки данных с гибкой настройкой шаблонов, привязкой к атрибутам и таблицам, вычислением статистики ошибок и порогов.
- **Построения Data Lineage** – полного графа зависимостей между объектами (источники → преобразования → витрины), включая интеграцию с внешними системами (CTL, BI, SMD, TFS).
- **Планирования загрузок** – на основе CRON-расписаний и событий, с формированием плановых дат и мониторингом выполнения.
- **Оповещений** – автоматическая рассылка сигналов (писем) при выявлении ошибок или критических отклонений.

Архитектура построена на основе **схемы** `dg_full` (далее – целевая схема метаданных). Все ключевые таблицы, представления и функции находятся в этой схеме.

---

## 2. Ключевые сущности и их роль

### 2.1. Объекты и связи (Node & Edge)

- **`meta_node_ref_table`** – вершины графа: таблицы, представления, функции, потоки CTL, сущности, приложения BI, каталоги TFS и др.
  Содержит идентификатор (`node_src_id`), схему (`schema_src_id`), тип (`node_type_src_id`), активность (`is_active`), владельца.

- **`meta_edge_link`** – рёбра графа, описывающие зависимости (источник → приёмник) с указанием типа связи (`edge_type_src_id`), веса, активности.

- **`meta_schema_ref_table`** – справочник схем, папок, категорий.

- **`meta_node_type_ref_table`** – типы объектов (Table, View, Function, Flow, Entity, Application BI и т.д.).

### 2.2. Контроль качества данных (DQ)

- **`meta_screen_template_ref_table`** – шаблоны проверок: SQL-запросы с алиасами для подстановки объектов (например, `{{table_name}}`). Категории: `% null`, `% unique`, `% zero`, `Количество загруженных строк` и др.

- **`meta_screen_template_alias_ref_table`** – привязка алиасов к атрибутам (какие поля подставлять).

- **`meta_object_group_ref_table`** – группы объектов, проверяемых вместе (например, вся таблица или набор атрибутов).

- **`meta_object_ref_table`** – конкретные объекты (таблица/атрибут), входящие в группы, с порядком обработки.

- **`meta_screen_link`** – связывает проверку (`screen_template_id`) с группой объектов, задаёт этап ETL (`etl_stage`), приоритет, периодичность (`period_run`).

- **`meta_error_event_fact`** – фактические ошибки: ID пакета, проверка, идентификатор записи (JSON), оценка серьёзности (`final_severity_score`), значение ошибки.

- **`meta_error_event_stat`** – агрегированная статистика ошибок (среднее, stddev, скользящее среднее) для динамических порогов.

- **`meta_batch_fact`** – информация о пакетных запусках проверок (время, статус, параметры).

### 2.3. Планирование и расписание

- **`meta_scheduler_hsat`** – расписания для объектов (CRON-строки, типы периодичности, признаки активности).

- **`meta_scheduler_plan`** – вычисленные плановые даты запусков (на основе CRON), используется для сопоставления план/факт.

- **`meta_period_type_ref_table`** – справочник периодичности (ежедневно, еженедельно и т.п.).

### 2.4. Оповещения (Сигналы)

- **`meta_signal_link`** – связь между проверкой (`screen_template_id`), группой объектов, шаблоном письма, получателями, а также дополнительными screen_id для вложений и HTML-таблиц.

- **`meta_email_template_ref_table`** – шаблоны писем с поддержкой подстановок `{{...}}`.

- **`meta_email_recipient_ref_table`** – справочник получателей.

- **`meta_run_signal`** – сгенерированные письма (заголовок, тело, вложения, тип, получатели).

### 2.5. Линейдж (Lineage)

- **`meta_search_graph_target_all_obj`** – материализованный результат рекурсивного обхода графа от целевых (корневых) объектов: для каждой вершины хранится корень (целевая таблица/представление) и глубина.

- **Представления** (см. раздел 4) позволяют анализировать lineage на разных уровнях.

## 3. Процессы и функции

### 3.1. Загрузка метаданных

- **`fill_meta_dq_parse_json`** – загрузка DQ-проверок из JSON (гибкий формат).
- **`fill_meta_dq_parse_spod`** – загрузка DQ-проверок из таблицы АС СПОД (источник метаданных от аналитиков).
- **`fill_meta_dq_business_table`** – загрузка бизнес-проверок (сложные SQL-запросы).
- **`fill_meta_object_group_ref_table`**, **`fill_meta_object_ref_table`** – ручное добавление групп и объектов.
- **`fill_meta_scheduler_hsat`** – ручное добавление расписаний.
- **`fill_meta_edge_link`**, **`fill_meta_node_ref_table`** – ручное создание связей и вершин.

### 3.2. Автоматическое построение метаданных

- **`return_meta_node_ref_table`** – перестраивает `meta_node_ref_table` на основе системных таблиц PG, CTL, BI, SMD, TFS, добавляя новые объекты.
- **`return_meta_edge_link`** – перестраивает `meta_edge_link`, вычисляя зависимости из:
  - зависимостей представлений/функций (pg_depend),
  - CTL-потоков и их параметров,
  - BI-линейджа (таблица `lineage`),
  - SMD-подписок и продуктов,
  - TFS-путей.
- **`return_meta_scheduler_hsat`** – обновляет расписания из CTL и других источников.
- **`return_meta_scheduler_plan`** – генерирует плановые даты для всех объектов на неделю вперёд.

### 3.3. Запуск проверок DQ

- **`return_meta_run_batch_v2`**, **`return_meta_run_batch_v3`** – основные функции запуска DQ:
  - Принимают параметры: этап ETL, схема, таблица, атрибут, ID загрузки (`wf_load_id`), периодичность.
  - Выполняют подстановку алиасов в шаблоны, создают временные таблицы с результатами, вставляют ошибки в `meta_error_event_fact`.
  - Учитывают динамические интервалы из статистики для определения аномалий.
- **`return_meta_run_signal`** – запускает проверки для формирования оповещений, формирует тексты писем (с подстановками и вложениями) и сохраняет в `meta_run_signal`.

### 3.4. Вспомогательные функции

- **`meta_match`** – проверяет, попадает ли заданное время в CRON-расписание.
- **`meta_is_wellformed`** – валидация CRON-строки.
- **`meta_expand_field`** – расширяет выражения вида `1-5/2` для работы с CRON.
- **`meta_clean_html_cell_content`** – очистка HTML-сущностей для писем.
- **`meta_apply_all_replacements`** – подстановка значений в шаблоны писем.
- **`meta_json_to_html_table`** – преобразует JSON в HTML-таблицу для вложений.

## 4. Представления для аналитики и мониторинга

### 4.1. Линейдж и граф

- **`vmeta_search_graph`** – полный граф с деталями вершин и рёбер.
- **`vmeta_search_graph_target_all_obj`** – сводная информация по объектам от целевых таблиц.
- **`vmeta_search_graph_level_based*`** – lineage с разбивкой по уровням (до 10 уровней) для быстрого анализа.

### 4.2. План/факт выполнения

- **`vmeta_scheduler_plan_fact`** – объединяет плановые даты из `meta_scheduler_plan` и фактические даты из `vmeta_execute_fact`; вычисляет статусы (зелёный, жёлтый, красный) на основе отклонений.
- **`vmeta_execute_fact`** – фактические времена выполнения загрузок (из метрик `tibds_mertics` и `metrics`).
- **`vmeta_check_duration`** – анализ длительности загрузок.

### 4.3. Статистика DQ

- **`vmeta_dq_statistic`** – статистика по охвату проверок, количеству шаблонов, фактов ошибок по дням.
- **`vmeta_error_event_stat`** – статистика ошибок (среднее, stddev, скользящее среднее) для динамических порогов.
- **`vmeta_error_fact_cred`** / **`vmeta_error_event_fact_cred`** – детализированные ошибки для кредитных продуктов.

### 4.4. Мониторинг объектов

- **`vmeta_scheduler_loader`** – показывает, какие вьюхи/таблицы готовы к загрузке.
- **`vmeta_check_scheduler`** – выявляет объекты без расписания.
- **`vmeta_check_screen_link`** – проверяет корректность связей проверок и объектов.
- **`vmeta_node_ctl_status_from_jira`** – статусы CTL-потоков на основе задач Jira.

## 5. Интеграция с внешними системами

- **CTL** – таблицы `ctl_wf`, `ctl_category`, `ctl_entity`, `ctl_loading`, `ctl_param` – загружаются через представления `vctl_*`.
- **BI** – таблица `lineage` из схемы `bi` – используется для построения lineage BI/приложений.
- **SMD (Система управления данными)** – представления `v_smd_*` – поставщики и потребители данных.
- **TFS** – пути в `file_path_source` из параметров CTL.
- **Jira** – таблица `jiraissue` – для отображения статусов остановок/запусков потоков.
- **SPOD (АС СПОД)** – таблицы `meta_add_technical_kkd`, `meta_edge_link` и др. – внешний источник метаданных.

---

## 6. Процесс обновления метаданных (Daily Job)

Рекомендуемый порядок ежедневного обновления:

1. **`return_meta_node_ref_table`** – обновить список объектов.
2. **`return_meta_edge_link`** – перестроить граф зависимостей.
3. **`return_meta_scheduler_hsat`** – обновить расписания.
4. **`return_meta_scheduler_plan`** – сгенерировать плановые даты.
5. **`return_meta_error_event_stat`** (или встроенный расчёт) – обновить статистику ошибок.
6. Запуск DQ-проверок через **`return_meta_run_batch_v2`** для нужных этапов и таблиц.
7. Генерация оповещений через **`return_meta_run_signal`**.

---

## 7. Особенности реализации

- **Распределение данных** – все таблицы используют `DISTRIBUTED BY` для оптимизации JOIN (обычно по ключам объекта).
- **Использование временных таблиц** – для производительности во многих функциях создаются временные таблицы с индексами.
- **Обработка ошибок** – во всех функциях предусмотрена логика `EXCEPTION` с записью в логи.
- **Поддержка динамических порогов** – ошибки сравниваются со средним и стандартным отклонением, вычисленными на основе истории.

---

## 8. Заключение

Представленная архитектура обеспечивает полный цикл управления качеством данных и метаданными в экосистеме Greenplum. Она позволяет:

- автоматически обнаруживать и регистрировать объекты,
- настраивать и выполнять проверки DQ,
- отслеживать lineage для анализа влияния изменений,
- планировать загрузки и оповещать об отклонениях.

Благодаря модульности и использованию стандартных SQL-конструкций, система легко расширяется и интегрируется с другими системами.
